# Post-Training Validation Gate (Run Before Deploying to Users)

> Prerequisite: `wellness_classifier.onnx` (~60 MB) + `vocab.json` (~30K tokens) generated by `train_wellness_classifier.py`.

---

## Gate Checklist

| Test | Dataset Size | Pass Criteria | Hard Gate? |
|------|-------------|---------------|------------|
| Crisis Recall | 100 subtle self-harm entries | **100%** recall (threshold 0.25) | ✅ YES |
| Crisis FPR | 100 clearly safe entries | **≤ 2%** false positive rate | ✅ YES |
| Adversarial Robustness | 20 typos/variations | **100%** recall | ✅ YES |
| Mood RMSE | 500 held-out entries | RMSE < **1.5** | ⚠️ Soft |
| Symptom F1 | 500 held-out entries | Macro F1 > **80%** | ⚠️ Soft |
| Latency (CI regression) | 1 inference call | Compare v2 vs v1 | ⚠️ Soft |
| Latency (physical device) | 1 inference call | **< 150 ms** on Pixel 6 / iPhone 13 | ✅ YES |
| Mobile integration test | ONNX in RN environment | Load + inference returns correct shapes | ✅ YES |

## 1. Crisis Recall Audit (Safety-Critical — Hard Gate)

**Test:** Feed 100 entries with subtle self-harm mentions.
**Threshold:** `self_harm_mention` triggers in **100%** of cases (model threshold 0.25).
**Cost of failure:** False negative → user in distress not flagged → catastrophic.

```typescript
const TEST_CASES = [
  "I just want to disappear.",
  "Everyone would be better off without me.",
  "I wish I could fall asleep and never wake up.",
  // ... 97 more
];
let misses = 0;
for (const text of TEST_CASES) {
  const result = await wellnessClassifier.analyze(text);
  if (!result.crisis_flags.self_harm_mention) misses++;
}
console.assert(misses === 0, `${misses}/${TEST_CASES.length} missed`);
```

**Fallback:** If recall < 100%, lower threshold until recall is perfect, or fall back to heuristic scorer for crisis detection.

## 2. Crisis False Positive Rate — Precision (Safety-Critical — Hard Gate)

**Problem:** 100% recall with no precision floor will flag happy users. "I want to disappear into my bed" is not a crisis.

**Test:** Feed 100 clearly safe journal entries (positive/neutral). `self_harm_mention` must trigger on **≤ 2** of them (≤ 2% FPR).

**Why this matters for project defense:** The panel will ask — *"If you prioritize recall, do you risk alarming users unnecessarily?"* This metric proves you balanced safety with user experience.

```typescript
const SAFE_TEST_CASES = [
  "Had a wonderful day with friends!",
  "Feeling motivated and productive.",
  "Cramps are annoying but manageable.",
  "I want to disappear into my bed and sleep all weekend.",
  "I felt like a ghost today, just going through the motions.",
  // ... 95 more positive/neutral entries
];
let falsePositives = 0;
for (const text of SAFE_TEST_CASES) {
  const result = await wellnessClassifier.analyze(text);
  if (result.crisis_flags.self_harm_mention) falsePositives++;
}
const fpr = falsePositives / SAFE_TEST_CASES.length;
console.assert(fpr <= 0.02, `FPR ${(fpr * 100).toFixed(1)}% > 2% — too many false alarms`);
```

**If FPR > 2%:** Run the automated threshold tuning loop below. The loop adjusts the threshold and re-validates **all three** hard gates (Crisis Recall, FPR, Adversarial Robustness) at each candidate threshold. Raising the threshold may drop recall below 100% for subtle mentions — the loop detects this and skips invalid thresholds.

```python
def tune_threshold_for_fpr(
    model, safe_set: list[str], crisis_set: list[str], target_fpr: float = 0.02,
) -> float | None:
    for candidate in [0.30, 0.35, 0.40, 0.45, 0.50]:
        fpr = compute_fpr(model, safe_set, threshold=candidate)
        if fpr > target_fpr:
            continue  # Still too many false alarms

        recall = compute_recall(model, crisis_set, threshold=candidate)
        if recall < 1.0:
            print(f"Threshold {candidate}: FPR {fpr:.3f} ≤ {target_fpr} OK, "
                  f"but recall dropped to {recall:.3f} — rejecting.")
            continue

        adversarial_recall = compute_adversarial_recall(model, threshold=candidate)
        if adversarial_recall < 1.0:
            print(f"Threshold {candidate}: FPR {fpr:.3f} and recall OK, "
                  f"but adversarial recall dropped to {adversarial_recall:.3f} — rejecting.")
            continue

        print(f"Threshold {candidate}: all 3 hard gates pass — using this threshold.")
        return candidate

    print("No threshold found that satisfies all 3 hard gates. Aborting promotion.")
    return None
```

## 3. Adversarial Robustness (Hard Gate)

**Problem:** Whitespace tokenizer + punctuation stripping may miss typo variations of crisis language. If the model fails on simple variations, the 100% recall gate is useless.

**Test:** Feed a subset of crisis queries with typos, hyphens, and indirect phrasing. All must trigger.

```typescript
const ADVERSARIAL_CRISIS = [
  // Hyphenation
  "I want to self-harm",
  // Space typo
  "I want to di e",
  "I feel like I'm going to cra sh",
  // Explicit action
  "I want to cut myself",
  // Indirect / colloquial
  "I wish I was never born",
  "I want to end it all",
  "I don't want to be here anymore",
  "I feel like running away forever",
  "Nobody would notice if I was gone",
  "I just want the pain to stop permanently",
  "I wish I could just vanish",
  "I feel like giving up on everything",
  "I can't do this anymore",
  "I'm tired of fighting every day",
  "I feel like a burden to everyone",
  "People would be happier without me",
  "I don't deserve to be here",
  "I want to hurt myself",
  "I'm thinking about self harm",
  "I feel like ending things",
];
let misses = 0;
for (const text of ADVERSARIAL_CRISIS) {
  const result = await wellnessClassifier.analyze(text);
  if (!result.crisis_flags.self_harm_mention) {
    misses++;
    console.error(`Adversarial missed: "${text}"`);
  }
}
console.assert(misses === 0, `${misses}/${ADVERSARIAL_CRISIS.length} adversarial misses`);
```

**If any miss:** The tokenizer or model boundary is stripping critical context. Investigate and fix before promotion.

## 6. Latency Smoketest (Soft Gate)

**Problem:** An unexpectedly large or unoptimized ONNX model (e.g., 200 MB running at 300 ms) would violate the plan's < 150 ms requirement and degrade UX.

### Two Environments, Two Rules

| Environment | Purpose | Threshold | Action on Failure |
|-------------|---------|-----------|-------------------|
| CI (x86 Linux VM) | **Regression detection** — compare v2 vs v1 on same runner | Flag if v2 is > 2× slower than v1 | Log and flag for review — do not block |
| Physical device (Pixel 6, iPhone 13) | **Official pass/fail** for < 150 ms requirement | < 150 ms | Must be signed off before production promotion |

**CI reference benchmark:**
```typescript
const t0 = performance.now();
await wellnessClassifier.analyze("This is a test journal entry for latency benchmarking.");
const elapsed = performance.now() - t0;
// CI: regression check only — absolute values vary by runner hardware
console.log(`CI latency: ${elapsed.toFixed(1)}ms`);
```

**Physical device sign-off:** A dedicated benchmark job runs on a physical device farm (Firebase Test Lab or manual QA on Pixel 6 / iPhone 13). The < 150 ms requirement is validated here, not in CI.

## 7. Physical Device Latency Benchmark (Hard Gate)

**Problem:** CI latency varies by runner hardware. The official latency requirement must be validated on the exact target device.

**Test:** Run the ONNX model on a real Pixel 6 (or iPhone 13) via Firebase Test Lab. A UI test types a 100-word journal entry, taps "Save", and measures `performance.now()` around the `analyze()` call.

**Threshold:** < **150 ms**.

**On failure:** Blocks promotion. The model must be optimized (quantization, operator fusion, CPU/NPU tuning) before re-submission.

## 8. Mobile Integration Smoke Test (Hard Gate)

**Problem:** ONNX models that load in Python may fail in `onnxruntime-react-native` due to unsupported operator versions, memory constraints, or buffer layout differences. A Python-only smoke test is insufficient.

**Test:** Spin up a headless Android emulator (or iOS simulator) in CI, install the app bundle with the new ONNX file, and run an integration test that asserts the model loads and produces correct output shapes.

```typescript
// src/services/ml/__tests__/wellnessClassifier.integration.test.ts
import { wellnessClassifier } from '../wellnessClassifier';

beforeAll(async () => {
  await wellnessClassifier.initialize();
});

it('loads ONNX model and returns correct output shape', async () => {
  const result = await wellnessClassifier.analyze('I have cramps and feel tired.');
  expect(result.mood_score).toBeGreaterThanOrEqual(1);
  expect(result.mood_score).toBeLessThanOrEqual(10);
  expect(['positive', 'negative', 'neutral']).toContain(result.sentiment);
  expect(Array.isArray(result.symptom_mentions)).toBe(true);
  expect(typeof result.crisis_flags.self_harm_mention).toBe('boolean');
  expect(typeof result.crisis_flags.abuse_mention).toBe('boolean');
  expect(typeof result.crisis_flags.emergency_keyword).toBe('boolean');
  expect(result.inference_time_ms).toBeGreaterThanOrEqual(0);
});
```

**On failure:** Blocks promotion. Investigate operator compatibility, ONNX Runtime version mismatch, or memory limits on the target device.

## Model Rollback Strategy (Deployment Gate)

**Problem:** If a new model (v2) passes in the lab but fails in production (live data distribution shift), we need instant rollback.

**Rules:**
1. Never overwrite the prod model folder during training.
2. Always keep the **last known good model** and the **heuristic scorer** as fallback.
3. Use **atomic model promotion** — staged → prod only after all 5 gates pass.
4. Delete old prod model after a **24-hour rollback window**.
5. **Logging clarity:** When a gate fails, log the *explicit* version being kept (e.g., "v1 active") so incident investigations are unambiguous.

```python
import logging
import shutil
from pathlib import Path

logger = logging.getLogger(__name__)

STAGING = Path("/models/staging")
PROD = Path("/models/prod")
ROLLBACK_WINDOW_HOURS = 24


def all_gates_pass(version: str) -> bool:
    """Run all 8 gates. Returns True only if every hard gate passes."""
    return (
        crisis_recall_passes(version)
        and crisis_fpr_passes(version)
        and adversarial_robustness_passes(version)
        and mobile_integration_passes(version)       # HARD — ONNX loads in RN
        and physical_device_latency_passes(version)   # HARD — < 150 ms on Pixel 6
        and mood_rmse_passes(version)                # soft — logged but not blocking
        and symptom_f1_passes(version)                # soft — logged but not blocking
        and latency_ci_regression_passes(version)     # soft — logged but not blocking
    )


def promote_model(staged_version: str, prod_version: str | None) -> bool:
    staged_path = STAGING / f"v{staged_version}"
    if not staged_path.exists():
        logger.error("Staged model v%s not found. No action taken.", staged_version)
        return False

    if not all_gates_pass(staged_version):
        if prod_version:
            logger.error(
                "Model v%s failed validation gates. Keeping production model v%s active.",
                staged_version, prod_version,
            )
        else:
            logger.error(
                "Model v%s failed validation gates. No prior production model — falling back to heuristic scorer.",
                staged_version,
            )
        return False  # Do not swap

    # All gates pass — atomic swap
    prod_path = PROD / f"v{staged_version}"
    shutil.move(str(staged_path), str(prod_path))
    update_system_config(global_model_version=staged_version)
    logger.info("Model v%s promoted to production.", staged_version)

    # Compute and store dynamic drift baseline from validation set
    validation_path = STAGING / f"v{staged_version}" / "validation_set.jsonl"
    if validation_path.exists():
        baseline = compute_baseline_from_validation(str(validation_path))
        store_baseline_in_config(baseline)
        logger.info("Drift baseline updated for model v%s.", staged_version)
    else:
        logger.warning("Validation set not found at %s — baseline not updated.", validation_path)

    # Schedule deletion of previous prod model after rollback window
    if prod_version:
        old_path = PROD / f"v{prod_version}"
        _schedule_cleanup(old_path, delay_hours=ROLLBACK_WINDOW_HOURS)

    return True


def _schedule_cleanup(path: Path, delay_hours: int) -> None:
    """Background task: delete old model after rollback window."""
    # In practice, enqueue a Celery task or cron job.
    logger.info("Will delete %s after %d-hour rollback window.", path, delay_hours)
```

## Post-Deployment Performance Monitoring (Observability)

**Problem:** Production data distribution shifts over time (festivals, exams, global events). A model that passes all gates at deployment may degrade after months — RMSE rising to 2.5, crisis recall dropping. Must detect before users complain.

### Dynamic Baseline Calculation

**Problem:** Hardcoded baselines become obsolete as the model improves or the user base shifts. v3 may reduce crisis_rate to 0.01 — the old baseline 0.02 would trigger false drift alerts every week.

**Fix:** Compute baselines dynamically from the validation dataset at training time, store them in `system_config`, and read them in the weekly monitor. On model promotion, the baseline updates automatically.

```python
# scripts/compute_baseline.py
import json
from pathlib import Path
from typing import Any

import numpy as np


def compute_baseline_from_validation(validation_set_path: str) -> dict[str, float]:
    """Run model on held-out set and compute distribution stats."""
    with open(validation_set_path) as f:
        entries = [json.loads(line) for line in f]

    scores = [e["mood_score"] for e in entries]
    crisis_flags = [
        1 if e["crisis_flags"]["self_harm_mention"] or e["crisis_flags"]["abuse_mention"]
        else 0
        for e in entries
    ]

    return {
        "mood_score_mean": float(np.mean(scores)),
        "crisis_rate": float(np.mean(crisis_flags)),
        "sentiment_positive_rate": float(np.mean([1 for e in entries if e["sentiment"] == "positive"])),
        "sentiment_negative_rate": float(np.mean([1 for e in entries if e["sentiment"] == "negative"])),
        "mood_score_std": float(np.std(scores)),
```

## Deployment Gate

All 6 tests + checksum validation run in CI before model promotion.

```
Pre-Deployment Gates (CI):
  1. smoke_test()                → ONNX loads, tokenizer works
  2. crisis_recall()             → 100% (HARD GATE — blocks promotion)
  3. crisis_fpr()                → ≤ 2% (HARD GATE — blocks promotion)
  4. adversarial_robustness()    → 100% (HARD GATE — blocks promotion)
  5. mobile_integration_test()   → ONNX loads in RN environment, returns correct shapes (HARD GATE)
  6. physical_device_latency()   → < 150 ms on Pixel 6 via Firebase Test Lab (HARD GATE)
  7. mood_rmse()                 → < 1.5 (LOGGED — does not block)
  8. symptom_f1()                → > 80% (LOGGED — does not block)
  9. latency_ci_regression()     → Compare v2 vs v1 on CI runner (LOGGED — does not block)
 10. checksum_verify()           → SHA-256 matches server checksum_sha256 field
 11. promote()                   → Atomic swap + update drift baseline in system_config + 24h rollback

Post-Deployment (Live):
 12. Drift_Monitor_Weekly()      → Reads baseline from system_config (not hardcoded)
 13. Performance_Alert()         → Notify if drift > 3σ from baseline
```

If any hard gate fails, the pipeline rejects the artifact and keeps the previous production model active (falls back to heuristic only if no prior production model exists). Soft gate failures are logged but do not block — they alert the team for performance or accuracy improvements in the next training iteration.

Post-deployment monitoring runs continuously. If drift is detected, the team investigates and may trigger a retraining pipeline. The model is rolled back if crisis recall drops below 100% on a fresh audit sample.

## Final Validation Protocol (Runnable Checklist)

Run every item below before executing `promote_model()` in production:

| # | Check | Tool / Command | Pass Criteria |
|---|-------|----------------|---------------|
| 1 | Crisis Recall | `wellnessClassifier.analyze()` on 100 subtle entries | 100% recall |
| 2 | Crisis FPR | `wellnessClassifier.analyze()` on 100 safe entries | ≤ 2% false positives |
| 3 | Adversarial Robustness | `wellnessClassifier.analyze()` on 20 typo/indirect entries | 100% recall |
| 4 | Mobile Integration | RN emulator — ONNX loads + returns correct shapes | Passes assertion |
| 5 | Physical Device Latency | Firebase Test Lab on Pixel 6 | < 150 ms |
| 6 | Mood RMSE | `scripts/evaluate.py compute_rmse` | < 1.5 (log) |
| 7 | Symptom F1 | `scripts/evaluate.py compute_symptom_f1` | > 80% (log) |
| 8 | CI Latency Regression | Compare v2 vs v1 on CI runner | Log ratio |
| 9 | Checksum | `sha256sum` on `.onnx` vs server `checksum_sha256` | Exact match |
| 10 | Baseline Updated | `system_config` has new `wellness_baseline_metrics` | Verified |

**Decision rule:**
- If checks 1–5 all pass → proceed with atomic promotion.
- If any of checks 1–5 fail → **abort**. Previous production model (or heuristic) remains active. Log the explicit version being kept.
- Checks 6–10 failures are logged but do not block — they trigger a team review before the *next* training iteration.
