## auth and onboarding
🔴 Gap 1: 5xx Error Handling in Hydration (Too Aggressive)
The Problem:

typescript
// Current behavior
if (GET /auth/me fails with 500) {
  clear tokens, clear cached user, set user: null // ← Forces logout
}
A 500 error from the server (e.g., database timeout, API crash) does not mean the token is invalid. Force-logging the user out on a temporary server error is hostile UX. The user will lose all offline progress and be forced to log in again, even though the server is just temporarily unavailable.

The Fix:

typescript
// In authStore.hydrate()
try {
  const user = await authService.getMe();
  setCachedUser(user);
  set({ user, isHydrated: true });
} catch (error) {
  if (isNetworkError(error)) {
    // Offline → fallback to cached user
    const cached = await getCachedUser();
    set({ user: cached, isHydrated: true });
  } else if (error.response?.status === 401 || error.response?.status === 403) {
    // Auth error → clear everything
    await tokenStore.clear();
    await clearCachedUser();
    set({ user: null, isHydrated: true });
  } else {
    // 5xx, 429, etc. → keep cached user, but flag as "stale"
    // Do NOT clear tokens or user. The app stays logged in.
    // A background refresh will retry later.
    const cached = await getCachedUser();
    set({ user: cached, isHydrated: true });
    // Optionally: show a subtle "Server unavailable" toast
    Toast.show({ type: 'warning', text1: 'Server unavailable. Using cached data.' });
  }
}
Why this matters: A temporary server outage should not forcibly log out thousands of users. This is a critical UX and trust issue.

🔴 Gap 2: Refresh Token Rotation in 401 Interceptor (Missing)
The Problem:
Your interceptor handles refresh, but if the refresh endpoint returns a new refresh token (as per your Phase 0 architecture with jti rotation), your interceptor must store it. Currently, it only appears to store the new access token:

typescript
// Current (likely)
const { data } = await axios.post('/auth/refresh', { refresh_token: refreshToken });
await tokenStore.setAccess(data.access_token);
// ❌ Missing: tokenStore.setRefresh(data.refresh_token) 
The Fix:

typescript
// In client.ts, refresh interceptor
const { data } = await axios.post('/auth/refresh', { refresh_token: refreshToken });
await tokenStore.setBoth(data.access_token, data.refresh_token); // ← Store BOTH
Why this matters: If you rotate refresh tokens (which you should—your user_sessions table uses jti), the old refresh token becomes invalid after rotation. If the interceptor doesn't store the new refresh token, the next refresh attempt will fail, and the user will be logged out.

The Problem:

The Fix:

Wrap all FlatList child components in React.memo.

Ensure props are stable (use useCallback for handlers passed to child).

typescript
// ✅ Correct
const JournalItem = React.memo(({ item, onPress }) => {
  // ... component
});

// In parent:
const renderItem = useCallback(({ item }) => (
  <JournalItem item={item} onPress={handlePress} />
), [handlePress]);
Why this matters: Without React.memo, the useCallback on renderItem is completely wasted. The child re-renders on every parent re-render, defeating the purpose of memoization.

🔴 Critical Gap 2: getItemLayout for Fixed-Height Items (Missing from Audit)
The Problem:
Your 17 FlatLists all use variable-height items (or at least React Native assumes they do). Without getItemLayout, React Native must measure every single item to calculate the scrollable area. This is expensive, especially on first render.

The Fix: Add getItemLayout for all FlatLists that have fixed-height items:

typescript
// For items with fixed height (e.g., 72px)
const getItemLayout = (data: any, index: number) => ({
  length: 72, // height of each item
  offset: 72 * index,
  index,
});

// Use in FlatList:
<FlatList
  data={data}
  renderItem={renderItem}
  getItemLayout={getItemLayout} // ← Optimizes scroll measurements
  // ... other props
/>
Why this matters: Without getItemLayout, React Native measures each item individually on scroll, causing jank. With it, the scroll engine calculates offsets instantly.

  renderItem={renderItem}

  // ...
/>

The Problem:

The Onboarding Flow Keyboard Logic:

Screen	Field	Return Key	Action on Submit

PersonalInfo	Height	"next"	Move focus to Weight
PersonalInfo	Weight	"done"	Dismiss keyboard, enable "Continue" button

PastCycle	Period Length	"done"	Dismiss keyboard
The Fix (Implementation Pattern):

typescript

const ageRef = useRef<TextInput>(null);
const heightRef = useRef<TextInput>(null);
const weightRef = useRef<TextInput>(null);

<FormField
  ref={ageRef}
  label="Age"
  returnKeyType="next"

<FormField
  ref={heightRef}
  label="Height (cm)"
  returnKeyType="next"
  onSubmitEditing={() => weightRef.current?.focus()}

    // Optionally auto-trigger continue
  }}

Why this matters:
Without focus chaining, a user entering 7 fields manually feels tedious and poorly optimized. This is a top 3 UX friction point in form-heavy onboarding flows.

🔴 Critical Gap 2: Android windowSoftInputMode & KeyboardAvoidingView Nuance
The Problem:
You noted that Android uses undefined behavior for KeyboardAvoidingView, which is typical. However, this is actually a trap in React Native.

On Android, KeyboardAvoidingView with behavior="padding" does not work reliably unless you explicitly set android:windowSoftInputMode="adjustResize" in AndroidManifest.xml.

Without this, the keyboard overlays the input instead of pushing the layout up.

The Fix (Add to android/app/src/main/AndroidManifest.xml):

xml
<activity
  android:name=".MainActivity"
  android:windowSoftInputMode="adjustResize"  <!-- ← CRITICAL -->
  android:configChanges="keyboard|keyboardHidden|orientation|screenSize|uiMode"
  android:launchMode="singleTask"
  android:exported="true">
Why this matters:
If adjustResize is missing, your KeyboardAvoidingView does nothing on Android, and the keyboard will cover the input field. This is a common crash-inducing bug for QA.
